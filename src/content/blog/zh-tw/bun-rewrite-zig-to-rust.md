2026 年 5 月 8 日，Bun 的創辦人 **Jarred Sumner** 發出了一個 Pull Request，標題只有五個字：**"Rewrite Bun in Rust"**。這個 PR 包含了 **6,755 個 commit**，涉及近百萬行程式碼的變更，而且整個重寫過程只花了大約六天。

這件事在程式設計社群引爆了一場關於 AI 輔助開發、語言選擇與工程哲學的巨大論戰。

## Bun 是什麼？

[Bun](https://github.com/oven-sh/bun) 是一個高效能的 JavaScript / TypeScript 執行環境（runtime），由 Oven 公司開發，目標是成為 **Node.js 的替代品**，同時也整合了套件管理（`bun install`）、打包工具（bundler）、測試框架（test runner）等功能，形成一個全面的 JavaScript 工具鏈。

**核心定位：**

- 比 Node.js 更快的啟動速度與執行效能
- 原生支援 TypeScript 與 JSX，無需額外設定
- 相容 Node.js API，讓多數現有專案無痛遷移
- 整合套件管理、測試、打包為單一工具

Bun 使用的 JavaScript 引擎是 **JavaScriptCore**（即 Safari 的引擎），而非 Node.js 使用的 V8。這是其效能優勢的重要來源之一。

2025 年底，Bun 被 **Anthropic**（Claude AI 的開發公司）收購，這為後來發生的一切埋下了伏筆。

## 那個 PR：六天重寫近百萬行程式碼

### 事件始末

[PR #30412](https://github.com/oven-sh/bun/pull/30412) 由 Jarred Sumner 在 2026 年 5 月 8 日提出，並在同一天合入主分支（對應的 commit 是 [`23427dbc`](https://github.com/oven-sh/bun/commit/23427dbc12fdcff30c23a96a3d6a66d62fdc091d)）。

PR 的描述相當簡短，核心內容如下：

> 通過了 Bun 在所有平台上現有的測試套件（並修復了數個記憶體洩漏問題和不穩定測試）。  
> 二進位檔案大小縮減了 3 MB 到 8 MB。  
> 基準測試結果介於持平到更快之間。  
> 最重要的是，**我們現在有了編譯器輔助的工具來捕捉與預防記憶體錯誤，這些錯誤多年來耗費了團隊大量的開發與除錯時間。**  
> 程式碼架構大致相同，相同的資料結構，仍然少用第三方函式庫，沒有 async Rust。

分支名稱是 `claude/phase-a-port`，清楚表明這次重寫大量使用了 **Anthropic 的 Claude AI agent** 來執行。

### 數字說話

- 約 **960,000 行 Zig 程式碼** 被移植為 Rust
- **6,755 個 commits**，涵蓋數千個檔案
- 完成時間：約 **6 天**
- 新 Rust 版本二進位檔縮小了 **3–8 MB**
- 所有平台現有測試：**全部通過**

## 好處：為什麼 Bun 要這樣做？

### 1. 編譯器強制的記憶體安全

這是 Jarred Sumner 在 PR 中點名的**最主要原因**。

Zig 雖然明確且高效，但它沒有類似 Rust 的借用檢查器（borrow checker）。多年來，Bun 的 Zig 程式碼累積了大量難以追蹤的**記憶體洩漏（memory leak）**問題。這些問題不只影響效能，也讓 Bun 在被 Anthropic 用作 Claude Code 的後端時，成為了一個穩定性隱患。

Rust 的型別系統在**編譯期**就能攔截大部分記憶體相關錯誤，讓這類 bug 從「上線後才會發現」變成「根本無法編譯通過」。

### 2. 更成熟的生態系與工具鏈

Rust 的生態系（`cargo`、`crates.io`、IDE 支援）遠比 Zig 成熟。更好的工具意味著更高的開發效率，也更容易吸引新的貢獻者加入。

### 3. AI 輔助開發的相容性

Anthropic 收購 Bun 之後，團隊轉向**以 AI agent 為核心的開發模式**。在這種工作流下，AI 更善於處理 Rust 的程式碼（有更多訓練資料、更成熟的型別錯誤訊息），而非仍在快速演進的 Zig。

### 4. 效能不退步，甚至更好

實際測試顯示，Rust 版本的效能介於持平到更快之間，並且還有優化空間（Jarred 在 PR 中提到仍有優化工作待完成）。

## 壞處：社群的質疑聲浪

### 1. 「Vibecoded Disaster」——13,000 個 `unsafe` 區塊

批評者最常提到的問題是：這份 AI 生成的 Rust 程式碼中，包含了超過 **13,000 個 `unsafe` 區塊**。

`unsafe` 在 Rust 中是一個特殊關鍵字，它告訴編譯器「這段程式碼我自己負責，你不要管我」——換句話說，它**繞過了 Rust 最核心的安全保證**。如果 `unsafe` 使用不當，等於把 Rust 退化成比 C 更危險的語言（因為讀者可能誤以為這段程式碼是安全的）。

大量的 `unsafe` 塊，加上 AI 自動生成的程式碼，讓人質疑這段程式碼實際上有多少人真正仔細審閱過。

### 2. 人工審閱的困境

6,755 個 commits 在幾天內是不可能被人工完整審閱的。這是 AI 大規模代碼遷移必然面對的問題：速度快，但**人類對程式碼的理解與掌控被大幅稀釋**。

### 3. 破壞了對 Zig 的承諾

Bun 一直是 Zig 生態系中**最知名的旗艦專案**之一。它的存在吸引了大量開發者嘗試 Zig，也為 Zig 在業界提供了有說服力的「生產級使用案例」。這次改寫等於是在告訴整個社群：就連最有名的 Zig 專案都放棄了 Zig。

## Zig 社群的 Beef：不只是技術問題

### 哲學上的根本分歧：AI 政策

Zig 官方維護了一項**嚴格的禁用 AI 政策**：禁止使用 AI 生成的 issue、PR 和留言。這項政策源自 Zig 社群對程式設計品質與人工責任感的高度重視。

而 Bun 被 Anthropic 收購後，走向了完全相反的方向——以 AI agent 為核心驅動整個開發流程。兩者的工程文化從根本上就是對立的。

### 技術上的積怨：被迫 Fork

在此次重寫事件之前，Bun 團隊就已經因為 Zig 官方不接受他們的效能優化補丁（例如平行代碼生成），而不得不**維護一個自己的 Zig fork**。這進一步增加了使用 Zig 的維護成本，也讓兩個社群的關係逐漸疏離。

### Andrew Kelley 的立場

Zig 的創辦人 Andrew Kelley 一直強調 Zig 仍是 pre-1.0 語言，用戶應該預期會有破壞性變更。這種「語言優先，用戶其次」的態度，雖然對 Zig 的長期品質有益，但對於需要在商業產品上快速迭代的團隊而言，無疑是一大痛點。

## 這件事的意義

Bun 的這次重寫，不只是一個公司的技術決策，它是 2026 年軟體工程現況的一個縮影：

1. **AI 已經能夠執行大規模的語言遷移任務**——雖然品質仍有爭議，但速度是人力無法比擬的。
2. **Zig 的生態系仍然脆弱**——失去 Bun 這個旗艦案例，對 Zig 的社群推廣是一個不小的打擊。
3. **Rust 仍是系統程式語言的主流選擇**——成熟的工具鏈、AI 友善的語法、強大的安全保證，讓它在與 Zig 的競爭中佔了上風。

## 結語

Bun 這個案例沒有絕對的對與錯。用 AI 快速遷移語言，得到了更穩定的記憶體安全保證，卻也留下了大量需要審閱的 `unsafe` 程式碼；且誰也沒辦法保證AI生成的代碼沒有任何問題。

沒有完美的技術選擇，只有在速度、安全、生態系與工程哲學之間，不斷權衡的取捨。
以及商業公司買下後，技術的走向不再是工程師說了算，而是商業考量。
