2026 年 5 月 8 日，Bun 的创始人 **Jarred Sumner** 发出了一个 Pull Request，标题只有五个字：**"Rewrite Bun in Rust"**。这个 PR 包含了 **6,755 个 commit**，涉及近百万行代码的变更，而且整个重写过程只花了大约六天。

这件事在程序员社区引爆了一场关于 AI 辅助开发、语言选择与工程哲学的巨大论战。

## Bun 是什么？

[Bun](https://github.com/oven-sh/bun) 是一个高效能的 JavaScript / TypeScript 运行环境（runtime），由 Oven 公司开发，目标是成为 **Node.js 的替代品**，同时也整合了包管理（`bun install`）、打包/构建工具（bundler）、测试框架（test runner）等功能，形成一个全面的 JavaScript 工具链。

**核心定位：**

- 比 Node.js 更快的启动速度与执行性能
- 原生支持 TypeScript 与 JSX，无需额外配置
- 兼容 Node.js API，让多数现有项目无痛迁移
- 整合包管理、测试、打包为单一工具

Bun 使用的 JavaScript 引擎是 **JavaScriptCore**（即 Safari 的引擎），而非 Node.js 使用的 V8。这是其性能优势的重要来源之一。

2025 年底，Bun 被 **Anthropic**（Claude AI 的开发公司）收购，这为后来发生的一切埋下了伏笔。

## 那个 PR：六天重写近百万行代码

### 事件始末

[PR #30412](https://github.com/oven-sh/bun/pull/30412) 由 Jarred Sumner 在 2026 年 5 月 8 日提出，并在同一天合并入主分支（对应的 commit 是 [`23427dbc`](https://github.com/oven-sh/bun/commit/23427dbc12fdcff30c23a96a3d6a66d62fdc091d)）。

PR 的描述相当简短，核心内容如下：

> 通过了 Bun 在所有平台上现有的测试套件（并修复了数个内存泄漏问题和不稳定性测试）。  
> 二进制文件体积缩减了 3 MB 到 8 MB。  
> 基准测试结果介于持平到更快之间。  
> 最重要的是，**我们现在有了编译器辅助的工具来捕捉与预防内存错误，这些错误多年来耗费了团队大量的开发与调试时间。**  
> 代码架构大致相同，相同的数据结构，仍然极少使用第三方库，没有 async Rust。

分支名称是 `claude/phase-a-port`，清楚表明这次重写大量使用了 **Anthropic 的 Claude AI agent** 来执行。

### 数据说话

- 约 **960,000 行 Zig 代码** 被移植为 Rust
- **6,755 个 commits**，涵盖数千个文件
- 完成时间：约 **6 天**
- 新 Rust 版本二进制文件缩小了 **3–8 MB**
- 所有平台现有测试：**全部通过**

## 好处：为什么 Bun 要这样做？

### 1. 编译器强制的内存安全

这是 Jarred Sumner 在 PR 中点名的**最主要原因**。

Zig 虽然明确且高效，但它没有类似 Rust 的借用检查器（borrow checker）。多年来，Bun 的 Zig 代码积累了大量难以追踪的**内存泄漏（memory leak）**问题。这些问题不只影响性能，也让 Bun 在被 Anthropic 用作 Claude Code 的后端时，成为了一个稳定性隐患。

Rust 的类型系统在**编译期**就能拦截大部分内存相关错误，让这类 bug 从“上线后才会发现”变成“根本无法编译通过”。

### 2. 更成熟的生态系统与工具链

Rust 的生态系统（`cargo`、`crates.io`、IDE 支持）远比 Zig 成熟。更好的工具意味着更高的开发效率，也更容易吸引新的贡献者加入。

### 3. AI 辅助开发的兼容性

Anthropic 收购 Bun 之后，团队转向**以 AI agent 为核心的开发模式**。在这种工作流下，AI 更善于处理 Rust 的代码（有更多训练数据、更成熟的类型错误提示），而非仍在快速演进的 Zig。

### 4. 性能不退步，甚至更好

实际测试显示，Rust 版本的性能介于持平到更快之间，并且还有优化空间（Jarred 在 PR 中提到仍有优化工作待完成）。

## 坏处：社区的质疑声浪

### 1. “Vibecoded Disaster”——13,000 个 `unsafe` 代码块

批评者最常提到的问题是：这份 AI 生成的 Rust 代码中，包含了超过 **13,000 个 `unsafe` 代码块**。

`unsafe` 在 Rust 中是一个特殊关键字，它告诉编译器“这段代码我自己负责，你不要管我”——换句话说，它**绕过了 Rust 最核心的安全保证**。如果 `unsafe` 使用不当，等于把 Rust 退化成比 C 更危险的语言（因为读者可能误以为这段代码是安全的）。

大量的 `unsafe` 代码块，加上 AI 自动生成的代码，让人质疑这段代码实际上有多少人真正仔细审阅过。

### 2. 人工审阅的困境

6,755 个 commits 在几天内是不可能被人工完整审阅的。这是 AI 大规模代码迁移必然面对的问题：速度极快，但**人类对代码的理解与掌控被大幅稀释**。

### 3. 破坏了对 Zig 的承诺

Bun 一直是 Zig 生态中**最知名的旗舰项目**之一。它的存在吸引了大量开发者尝试 Zig，也为 Zig 在业界提供了有说服力的“生产级使用案例”。这次改写等于是在告诉整个社区：就连最有名的 Zig 项目都放弃了 Zig。

## Zig 社区的 Beef：不只是技术问题

### 哲学上的根本分歧：AI 政策

Zig 官方维护了一项**严格的禁用 AI 政策**：禁止使用 AI 生成的 issue、PR 和留言。这项政策源自 Zig 社区对编程质量与人工责任感的高度重视。

而 Bun 被 Anthropic 收购后，走向了完全相反的方向——以 AI agent 为核心驱动整个开发流程。两者的工程文化从根本上就是对立的。

### 技术上的积怨：被迫 Fork

在此次重写事件之前，Bun 团队就已经因为 Zig 官方不接受他们的性能优化补丁（例如并发代码生成），而不得不**维护一個自己的 Zig fork**。这进一步增加了使用 Zig 的维护成本，也让两个社区的关系逐渐疏离。

### Andrew Kelley 的立场

Zig 的创始人 Andrew Kelley 一直强调 Zig 仍是 pre-1.0 语言，用户应该预期会有破坏性变更。这种“语言优先，用户其次”的态度，虽然对 Zig 的长期质量有益，但对于需要在商业产品上快速迭代的团队而言，无疑是一大痛点。

## 这件事的意义

Bun 的这次重写，不仅是一个公司的技术决策，它是 2026 年软件工程现状的一个缩影：

1. **AI 已经能够执行大规模的语言迁移任务**——虽然质量仍有争议，但速度是人力无法比拟的。
2. **Zig 的生态系统仍然脆弱**——失去 Bun 这个旗舰案例，对 Zig 的社区推广是一个不小的打击。
3. **Rust 仍是系统级编程语言的主流选择**——成熟的工具链、AI 友好的语法、强大的安全保证，让它在与 Zig 的竞争中占了上风。

## 结语

Bun 这个案例没有绝对的对与错。用 AI 快速迁移语言，得到了更稳定的内存安全保证，却也留下了大量需要审阅的 `unsafe` 代码；且谁也没办法保证 AI 生成的代码没有任何潜在隐患。

没有完美的技术选择，只有在速度、安全、生态系统与工程哲学之间，不断权衡的取舍。
以及商业公司买下后，技术的走向不再是工程师说了算，而是商业考量。
